# インタラクション・情報デザイン原則

ユーザーの役割を定義した後、OMTA 固有の実装詳細を選択する前に、これらの原則を適用する。

## 意思決定デザイン

- 配置したいコンポーネントからではなく、次の正しいユーザーの意思決定から始める。
- サーフェスごとに1つの明確なプライマリアクションを優先する。複数のアクションが存在する場合、状態に応じてプロモート、デモート、マージ、延期、またはゲートするかを決定する。
- 破壊的または高コストなアクションは、安全なアクションよりも後に表示し、より強い結果シグナリングを付与する。

## 情報デザイン

- 最も意思決定に関連する情報を、視線が最初に到達する場所に配置する。
- リスタイルする前に、コンテンツを削除、グループ化、順序付け、要約、または段階的に開示する。
- 想起よりも認識を優先する: ラベル、例、デフォルト値、可視的なコンテキストは隠されたルールに勝る。

## 密度とコピー

- 読解努力をコストとして扱う。余分な一文ごとに意思決定または信頼構築の目的が必要。
- 見出しはオリエンテーション、ヘルパーテキストはアクションに不可欠な説明、エラーテキストはリカバリに使用する。これらの役割を混在させない。
- まず要約し、根拠、例外、詳細は関連性が生じた時にのみ開示する。

## 状態とフィードバック

- ユーザーがコミットする前にステータスを可視化する: 現在の選択、未保存の変更、バリデーション状態、不可逆な結果、アクション後の成果。
- すべての非同期またはデータ依存サーフェスには、ローディング、空、成功、エラー、権限の明示的な動作が必要。
- フィードバックは3つの質問に素早く答えるべき: 何が起きたか、それは何を意味するか、次に何をすべきか。

## フォームとワークフロー

- バックエンドモデルの形状ではなく、ユーザーの意図に沿ってフォームをシーケンスする。
- 現在のステップで必要な情報のみを尋ねる。
- 有用な範囲で早期にバリデーションするが、低リスク入力に対するノイジーな中断は避ける。
- 高コストなミスは制約、プレビュー、確認、またはアンドゥで防止すべき。

## 体系化

- ソリューションが繰り返される場合、一回限りの修正をコピーするのではなくパターンを標準化する。
- 新しい状態、ロール、ロケールをレイアウト崩壊なしに吸収できる構成可能な構造を優先する。

## 症状マッピング

- 複数の送信/保存ボタンは通常 `アクション階層` の問題を示す。
- テキスト過多の画面は通常 `情報階層` と `認知負荷管理` の問題を示す。
- 隠された副作用は通常 `状態の可読性` の問題を示す。
- ハッピーパスのみのフローは通常 `エラー防止とリカバリ` および `拡張性` の問題を示す。
