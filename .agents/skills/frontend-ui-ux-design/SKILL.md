# フロントエンド UI/UX デザイン

フロントエンドの品質が抽象的なプロダクト UX の判断と OMTA 固有の実装契約の両方に依存する場合にこのスキルを使用する。

## ノーススター

優れた SaaS サーフェスはユーザーに以下を可能にする:

- 自分がどこにいるか、何が重要か、システムがどの状態にあるかを理解する
- 最善の次のアクションを素早く特定する
- そのアクションを低摩擦・低リスクで実行する
- 何か問題が起きた時に安全にリカバリする
- 将来の状態をサーフェスの再設計なしに対応する

## ワークフロー

1. UI を変更する前にユーザーの仕事を定義する。
   - `references/methodology-review-axes.md` を読む。
   - ユーザーの目標、重要な判断、失敗コスト、成功シグナルを1〜2文で述べる。
2. インタラクションと情報デザインの原則でサーフェスを診断する。
   - `references/interaction-and-information-principles.md` を読む。
   - `目的の明確性`、`アクション階層`、`情報階層`、`認知負荷管理`、`段階的開示`、`状態の可読性`、`エラー防止とリカバリ`、`一貫性とシステム活用`、`アクセシビリティと耐障害性`、`拡張性` の観点で問題を記述する。
3. ソリューションを OMTA の正規オーナーサーフェスにマッピングする。
   - `references/ownership-and-surface-selection.md` を読む。
4. ソリューションを OMTA のデザインシステムとコピー契約で表現する。
   - `references/design-system-and-interaction-rules.md` を読む。
5. 変更が本当に必要とするゲートのみをバリデーションする。
   - `references/validation-matrix.md` を読む。
6. 第一原理の順序でレビューまたは所見を報告する。
   - 目的またはアクション階層の問題
   - 情報階層または認知負荷の問題
   - 状態、リカバリ、またはアクセシビリティの問題
   - オーナー、境界、またはデザインシステムの問題
   - バリデーションカバレッジの欠落

## コア非交渉事項

- サーフェスごとに1つの明確なプライマリアクションを優先する。複数のアクションが避けられない場合は、セカンダリアクションをデモート、グループ化、シーケンス、または延期して階層を明示する。
- 次の正しいアクションを取るために必要な読解量を最小化する。
- すべてのオプション、エッジケース、説明を前面に出すのではなく、複雑さを段階的に開示する。
- コミット前に現在の状態、結果、リカバリを可視化する。
- 新しいインタラクションモデルを発明する前に、確立されたパターンを再利用する。

## OMTA 非交渉事項

- `apps/*` 配下に共有ビジネスロジックを追加しない。
- Web コードに `@omta/*/server` をインポートしない。
- 重複するトークン、テーマ、プリミティブの実装で `@omta/ui` をバイパスしない。
- `apps/app` で、Web ローカルコンテキストコンポーネントを `src/components/ui` に配置しない。プリミティブガバナンス契約に別段の定めがない限り `src/components` に保持する。
- i18n キーが管理すべきインラインのユーザー向けコピーを追加しない。
- 場当たり的なカラー、z-index、ブラー値、タイポグラフィ値、`transition: all` を導入しない。
- タスクが明示的に OMTA のデザイン SoT を更新する場合、またはユーザーがその転換を明示的に要求する場合を除き、新しいプロダクトビジュアル言語を発明しない。
