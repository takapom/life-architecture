# DAG 分割ルールの登録

## ノード設計
- レビュー可能な機能または API 境界で分割する。
- 可能な限り、同一ファイルの編集は1つのノードにまとめる。
- 明示的な依存関係なしにクロスカッティングなノードを避ける。
- 衝突を減らすために `allowed_files` を狭くする。
- `Commit Units` をレビュー可能なアトミックな変更として定義する（各ユニットが単独で理解可能であるべき）。
- 1つのノードで `Commit Units` をアトミックに保てない場合、実装前にノードを分割するかユニットを書き直す。

## 依存関係ルール
- B が A の成果物を使用する場合、`A -> B` を設定する。
- 2つのノードが同じファイルに触れる場合、マージまたは直列化する。
- ノードが共有規約（API/スキーマ/コピー）を更新する場合、下流の依存関係を追加する。
- ノード内のユニット順序を明示的にする（`CU1 -> CU2 -> ...`）。コミットケイデンスが同じシーケンスに従えるようにする。

## 出力規約
- すべてのノードに必須: `id`、`branch`、`allowed_files`、`acceptance_checks`、`tests`。
- すべてのアクション可能なソースアイテムは `covers` または `deferred_items` に含まれる必要がある。
- `issue_tracking.strategy=remote-github-sot` の場合、`issue_map` とノードの `github_issue` マッピングを提供する。
- 各タスク Issue に `Commit Units` セクションを追加し、PR のコミット履歴がユニット順序に従うようにする。
