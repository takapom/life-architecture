# Playwright インタラクティブ

ローカル Web アプリや Electron アプリに対して、変更のたびにブラウザハンドルを破棄することなく高速にイテレーションしたい場合に、`js_repl` で永続的な Playwright セッションを使用する。

## 使用場面

- ローカル UI に対して再現可能な機能・ビジュアル QA が必要な場合。
- 複数の修正にわたって同じブラウザ、コンテキスト、または Electron ウィンドウを維持したい場合。
- 変更がリロードだけで済むのか、フルリランチが必要なのかを判断する必要がある場合。

## 前提条件

- このスキルを使用する前にセッションで `js_repl` を有効化し、ツールリストが更新されていない場合は Codex を再起動する。
- デバッグ対象のアプリと同じワークスペースから実行する。
- ヘッド付き Playwright または Electron アクセスが必要な場合は `danger-full-access` を使用する。
- ターゲットワークスペースに `playwright` をインストールし、ターゲットが Electron アプリの場合のみ `electron` をインストールする。
- `js_repl_reset` は再利用しようとしている永続ハンドルを破壊するため、リカバリ専用として扱う。

## 手順

1. Playwright を操作する前に QA インベントリを作成する。
   - 要求された振る舞い、変更したユーザー向けコントロールや状態、最終回答で述べる予定の主張を記録する。
   - 脆弱な振る舞いを露呈しうるハッピーパス外のチェックを少なくとも2つ追加する。
2. ブートストラップセルを一度実行し、共有ハンドルをトップレベルの `var` バインディングで保持する。
   - ブートストラップ、Web/Electron セッションスニペット、スクリーンショット、クリーンアップについては [セットアップとレシピ](references/setup-and-recipes.md) を参照。
3. ナビゲーション前に、永続 TTY セッションで開発サーバーを起動または確認する。
   - ローカル URL には `localhost` より `127.0.0.1` を優先する。
4. 編集後も同じ Playwright ハンドルを再利用する。
   - レンダラーのみの変更にはリロード。
   - スタートアップ、Electron メインプロセス、またはプロセス所有権の変更後はリランチ。
5. まず機能 QA を実行し、次にビジュアル QA を実行する。
   - ビューポートフィットチェック、エビデンス要件、よくある不具合については [QA と障害モード](references/qa-and-failure-modes.md) を参照。
6. 最終回答の各主張を裏付けるスクリーンショットと観察結果を取得する。
7. Playwright セッションはタスクが実際に完了した時にのみ閉じる。
