# デザイン思想

## コンセプト

**「診断ツールではなく、鏡を渡す体験」**

このアプリはクイズでも占いでもない。
ユーザーが自分の人生をエンジニアリングの言語で語れるようになる体験を設計する。

結果ページのアーキテクチャ図が「自分だ」と感じられること——それが唯一の成功基準。

---

## デザインレイヤー構造

```
┌─────────────────────────────────────────┐
│  Layer 2: Terminal Aesthetic             │  ← エンジニア向けの質感・文体
│  (Fira Code / dark bg / green accent)   │
├─────────────────────────────────────────┤
│  Layer 1: Zenn-inspired Base             │  ← 読みやすさ・余白・情報設計
│  (余白 / タイポグラフィ / コンテンツ主役)  │
└─────────────────────────────────────────┘
```

**Layer 1（Zenn）** がレイアウト・可読性・情報設計の土台を担う。
**Layer 2（Terminal）** がエンジニアとしての「自分ごと感」を演出する表面テクスチャになる。

ターミナル風はあくまで「味付け」であり、「テーマ」ではない。
読みにくい黒背景に緑文字で埋め尽くすのではなく、Zennの清潔さの上にターミナルの質感を薄く乗せる。

## デザイン原則

### 1. コンテンツ・ファースト、クローム・ミニマル（Zenn層）

UIは自己主張しない。ユーザーの回答と診断結果が常に主役。

ナビゲーション・ボタン・装飾は必要最小限に絞る。
画面に余白があることへの恐れを持たない。

> Zennが記事を主役にするように、このアプリは「あなたの人生の言語化」を主役にする。

### 2. テクニカル・オーセンティシティ（Terminal層）

「エンジニア向けアプリっぽいデザイン」ではなく、エンジニアが実際に使う道具の質感を持つ。

ターミナルの表現は以下の箇所に限定して使う——全面に敷くのではなく、文脈がある場所に置く:

- アーキテクチャ名（`Tennis-Centric Monolith` のようなラベル表示）
- ローディング画面の「解析ログ」風テキスト
- 診断結果の構成図ノードラベル
- コードブロック的なUI要素（スコア表示など）

### 3. プログレッシブ・リビール

診断フローは「答えを隠して最後に渡す」構造。この緊張と解放をUIで増幅する。

- 質問は1問ずつ。全体像は見せない
- ローディング画面は「解析中」の演出として機能させる
- 結果ページは情報を一気に見せず、アーキテクチャ名→図→スコア→コメントの順で展開する

### 4. タイポグラフィ・ハイアラーキー（Zenn層 × Terminal層）

| 用途 | フォント | 役割 |
|------|----------|------|
| 見出し・アーキテクチャ名 | **Fira Code** | ターミナル質感・キャッチーさ |
| 本文・説明文・AIコメント | **Fira Sans** | 可読性・清潔感 |
| UI補助テキスト（ラベル・キャプション） | Fira Sans Light | 控えめな補足 |

Fira Code × Fira Sansは同一ファミリー設計のため、両方並べても統一感が崩れない。

本文の行間は1.7〜1.8、1行あたり38〜60文字を目安にする。

### 5. カラー：意味を持つ、少なく使う

```
Background  #020617   ← ディープネイビー（純黒を避けることで柔らかさを残す）
Surface     #0F172A   ← カード・パネルの背景
Muted       #1E293B   ← セパレーター・ボーダー
Text        #F8FAFC   ← 主要テキスト
Accent      #22C55E   ← CTA・選択状態・強調（Terminal の緑を意味信号として使う）
```

- アクセントの緑（`#22C55E`）はターミナルの出力色から取った。しかし「装飾の緑」ではなく「意味の緑」として使う
- 主要CTA・選択状態・スコアの「高」に限定して使う
- 背景はZennの白地ではなくディープネイビーを採用。Zennの余白設計をダークモードに移植した形

### 6. モーション：意味のある動きだけ

トランジションはユーザーの状態遷移を補助するためにある。演出のためではない。

- 質問から質問への遷移：スライドで「次へ進む」感（200〜300ms）
- ローディング：「処理が走っている」ターミナル風ログ流し
- 結果の展開：フェードインで情報を順番に渡す

アニメーションは `prefers-reduced-motion` を尊重する。
`transform` / `opacity` を使い、`width` / `height` によるアニメーションは避ける。

## スクリーン別の設計意図

### ランディング

**目標**: 「面白そう」と「自分も診断できる」を3秒で伝える。

- アーキテクチャ図のプレビューを見せる（結果の期待値を上げる）
- Fira Codeでキャッチコピーを表示し、診断ツールの質感を最初から出す
- GitHubログインボタンが唯一のCTA（アクセントグリーン）

### 診断フロー（質問画面）

**目標**: 回答に集中させる。離脱させない。

- 質問文を画面の中心に大きく（Fira Sans 24px以上）
- プログレスバーは左から埋まるアクセントグリーン
- テキストエリアは広め、`border: 1px solid var(--color-border)` で控えめな枠線
- 「次へ」ボタンは回答が入力されたら有効化（グレー→アクセントグリーン）

### ローディング

**目標**: 待ち時間をワクワクに変える。

- ターミナルの最も強い演出場面。Fira Codeで「ログ」が流れる
- `Analyzing architecture patterns...` のような技術的言語
- キーワードが回答内容から拾われて流れる（本当にAIが動いている感）

```
// ローディング画面のテキスト例
> Parsing input vectors...
> Detecting coupling patterns...
> Generating architecture label...
> Solo-Optimized Batch Processor — matched
```

### 診断結果

**目標**: 「これは自分だ」と感じさせる。シェアしたいと思わせる。

情報の展開順：

```
1. アーキテクチャ名（Fira Code・大きく・アクセントグリーンで強調）
2. 静的フローボード + 代表アクセスフロー（入口から処理順が読める）
3. 特性スコア（レーダーチャート）
4. AI解説コメント（Fira Sans・メタファーを交えた文章）
5. 「過去も診断する」CTA
```

- 構成図のノードラベルはFira Code。エンジニアが読む設計書の質感
- 構成図は依存関係図で終わらせず、代表アクセスを選ぶと処理順が追えるようにする
- AIコメントはFira Sansで書く。ターミナルではなく「所見文書」のトーン

### 変遷図（タイムライン）

**目標**: 過去と現在の差分を「成長」として読めるようにする。

- 2つの構成図を並べてdiffを視覚化
- アーキテクチャ名の変化がFira Codeで並ぶことで「バージョン比較」の質感が出る

## Zennから借りるもの・借りないもの

### 借りるもの（精神・アプローチ）

- **コンテンツを主役にする設計姿勢**
- **余白を恐れない**
- **タイポグラフィで情報ヒエラルキーを作る**
- **シンプルなナビゲーション**

### 借りないもの（差別化・上書き）

| Zenn | このアプリ |
|------|-----------|
| 白背景・ライトモード | ディープネイビー背景（ダークモード固定） |
| ブランドカラー（緑）を装飾に使う | アクセントグリーンを「意味の信号」に限定 |
| ゴシック体の本文 | Fira Sans（エンジニアリング系のサンセリフ） |
| 静的な読み物体験 | ターミナルログ演出のある診断フロー |

---

## やらないこと

- 純黒背景（`#000000`）は使わない。`#020617` のディープネイビーで柔らかさを残す
- 緑のマトリックス降下演出（ターミナルのステレオタイプ）
- ゲーミフィケーション的な過剰演出（バッジ・ポイント・レベルアップ）
- 多色使い（カラーパレットは上記6色に固定）
- スクロールアニメーション過多
