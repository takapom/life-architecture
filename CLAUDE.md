# CLAUDE.md — 人生アーキテクチャ診断

## プロジェクト概要

ユーザーの人生をソフトウェアアーキテクチャに見立てて診断・可視化するエンタメ型自己分析Webアプリ。
ターゲットはソフトウェアエンジニア。

詳細は `docs/` を参照。
- 要件定義: `docs/requirements.md`
- 技術アーキテクチャ: `docs/architecture.md`
- オーケストレーション手順: `docs/orchestration-workflow.md`
- デザイン思想： `docs/design.md`

---

## 技術スタック・アーキテクチャ方針
`docs/architecture.md` を参照。
(必要な場合のみ参照)

### やってはいけないこと

- `ai/interface.ts` のインターフェースを破壊的変更しない
- React Flow のノード/エッジをユーザーが編集できる設定にしない（読み取り専用を維持）
- モジュールをまたいだ直接インポートを増やさない（モジュール境界を守る）
- AIプロバイダーを切り替える場合は `providers/` に実装を追加し、`interface.ts` は変更しない

## 開発ルール

### 実装前に計画を出す

3ステップ以上のタスクはいきなり実装せず、先に方針を提示してユーザーの確認を取る。

### 設計判断は複数案を比較する

設計に関わる変更では、力技の前に2〜3のアプローチを比較検討する。些細な修正には適用しない。

### 完了前に品質を自問する

実装完了前に「スタッフエンジニアがレビューを承認できるレベルか」を自問する。ビルドが通ることは最低条件。

### バグは自律調査してから報告する

バグ報告を受けたら、ユーザーに聞く前にまず自分で原因を調査して修正案を出す。設計判断が必要な場合のみ確認を取る。

### コンテキストを汚染しない

リサーチや大規模な調査はサブエージェントに投げ、メインのコンテキストウィンドウを節約する。

### ミスパターンを記録する

実装で犯したミスや修正パターンは `docs/revision_log.md` に追記する。次のセッションで参照して同じミスを繰り返さない。

### 再利用できるものは必ず共通化する

同じロジック・型定義・コンポーネント・ユーティリティが複数箇所に散在する状態を放置しない。
実装時に以下を意識する：

- **型定義** は `types/` に集約し、各モジュールが参照する形にする
- **ユーティリティ関数** は `lib/` または `utils/` にまとめ、同等の処理を別々に書かない
- **UIコンポーネント** は共通部品として切り出し、ページ固有のロジックと分離する
- **定数・設定値** は一箇所で管理し、マジックナンバーや文字列を散在させない
- 新しい実装を書く前に「既存コードに同等のものがないか」を必ず確認する
- コピーペーストで実装を済ませることは原則禁止。抽象化して共通化する
